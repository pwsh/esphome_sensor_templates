# Architecture

How this library is put together and why. The authoring rules themselves live in
[CONVENTIONS.md](CONVENTIONS.md); this document explains the system around them.

## The template file model

Every file under `templates/` is a **self-contained ESPHome package**: one include gives a
working, documented feature with sensible defaults. Files never reference ids defined in other
template files — when an entity needs a source (WiFi quality needs RSSI), the file carries its
own `internal: true` source sensor. Top-level keys (`sensor:`, `debug:`, `wifi:`, `esphome:`)
deep-merge across packages, which is what lets dozens of includes coexist in one device config.

Three usage tiers, all equivalent:

```yaml
packages:
  uptime: !include esphome_sensor_templates/templates/diagnostics/uptime.yaml   # zero-config
  wifi: !include                                                                # per-sensor vars
    file: esphome_sensor_templates/templates/network/wifi_signal.yaml
    vars: { st_update_interval: 5min }
  heap: github://pwsh/esphome_sensor_templates/templates/diagnostics/heap.yaml@main  # remote
```

## Substitution semantics (verified, not assumed)

ESPHome gives package files two variable mechanisms with **different precedence**, verified
empirically on 2026.6.5 (the docs are misleading here):

| Mechanism | Resolution | Overridden by |
|---|---|---|
| `defaults:` | at include time, file-local | per-include `vars:` only |
| package `substitutions:` | merged into the global table | user's top-level `substitutions:`, and `vars:` per file |

Effective order: CLI `-s` → include `vars:` → file `defaults:` → user top-level
`substitutions:` → package `substitutions:`.

The library exploits both deliberately:

- **Shared knobs** (`st_update_interval`, `st_state_class`, `st_name_prefix`,
  `st_disabled_by_default`, `st_internal`) live in every file's `substitutions:` block at
  identical values — so one user substitution overrides the whole library, while `vars:` still
  wins per include. The generator enforces value uniformity because these blocks merge globally.
- **File-specific vars** (`st_ntp_server`, pins, thresholds) live in `defaults:` — stable
  regardless of what the user names their own substitutions.
- **Required vars** (secrets, pins, device name) appear in neither block, so they resolve from
  `vars:` *or* a top-level substitution — which is exactly how the web builder's hoisting and
  `device_base` work with no `vars:` blocks at all.

## Shared hardware resources

- `core/i2c.yaml` owns the I2C bus (`st_i2c_bus`); I2C sensors point at it via `*_i2c_id` vars.
- UART devices (LD2410/LD2450, GPS) each own a `uart:` entry with unique ids and pin vars.
- I2S microphone and speaker own separate buses (single-I2S chips can run only one).
- Cross-package attachment uses `!extend` with a var-driven id (`daily_restart` schedules onto
  `${st_daily_restart_time_id}`, default `st_time_sntp`) — plain re-declaration of an id is a
  hard "ID redefined!" error, and ESPHome permits only one SNTP instance.

## Metadata pipeline

Each template starts with a machine-parsed header (`# @template / @title / @category /
@description / @platforms / @requires / @entities / @var name [default] {choices}: … / @note`).
That header is the **single source of truth** for three consumers, all generated by
`tools/build_catalog.py` (stdlib-only, deterministic, idempotent):

```
templates/*/*.yaml headers
        │
        ▼
tools/build_catalog.py ──► web/catalog.json          (machine catalog incl. file bodies)
        ├────────────────► docs/<slug>.md            (one page per template, usage snippets)
        ├────────────────► README.md index           (between the GENERATED INDEX markers)
        └────────────────► web/index.html            (catalog injected into <script id="catalog">)
```

The generator also validates: slug/filename/category agreement, `@var` ↔ `defaults:`/
`substitutions:` reconciliation, required-vars-absent-from-defaults, shared-knob value
uniformity, enum defaults ∈ choices, and library-wide id uniqueness.

## Web builder (`web/index.html`)

A single self-contained page (inline CSS/JS, no frameworks, no external requests) served from
GitHub Pages and equally functional from `file://`. The catalog is embedded at build time with a
`fetch('catalog.json')` fallback.

- **Device panel** — board picker (10 generic per-variant entries emitting the recommended
  `variant:`-only form + 13 named boards + custom), esphome identity fields (name,
  friendly_name, comment, area, MAC suffix, project), network (DHCP/static IP, WiFi secrets,
  fallback-AP SSID, NTP server), IANA timezone dropdown (`Intl.supportedValuesOf`). No-WiFi
  chips (H2, P4) suppress the WiFi surface entirely.
- **Three output modes** — remote `github://` packages, local vendored includes, and inline YAML
  (an indentation-based structural merger that handles block scalars, key conflicts, and
  timezone injection). All modes share one plan builder.
- **Substitutions hoisting** — customizations (identity, network, unique template vars) are
  collected into one commented `substitutions:` block; required vars hoist cleanly, defaulted
  vars use pass-through `vars: { x: "${x}" }` entries (required by the verified precedence).
- **Requirements advisor** — `@requires` values normalize to a satisfied-map (wifi/api/time/
  i2c/microphone/psram) with one-click add-template fixes; free-text requirements surface as
  reminders. Chip capability aware (wifi unsatisfiable on H2/P4).
- **Multi-instance** — templates whose entity id is itself a var (detected from the catalog, not
  hardcoded) can be added repeatedly with auto-suffixed ids/names.
- The pure generator functions are exported (`module.exports`) and exercised by a Node smoke
  harness; the acceptance bar for every builder change is that its generated configs pass
  `esphome config`.

## Verification strategy

Three layers, in increasing strength:

1. `tools/build_catalog.py` — header/convention validation on every run.
2. `tools/validate.sh` — `esphome config` over `examples/` (a venv is created and dummy
   secrets generated on first run). `examples/all_templates.yaml` merges every S3-compatible
   template at once — the no-collision proof; `examples/peripherals_esp32.yaml` covers the
   classic-ESP32 pin-map-heavy set and sources its whole `esphome:` block from `device_base`.
3. Real firmware compiles (`esphome compile`) for lambda-bearing and codegen-heavy template
   sets — the only layer that catches missing C headers (config validation happily passes
   configs that cannot build). IDF headers are force-included per file via
   `platformio_options: build_src_flags`.

## Deployment

`.github/workflows/pages.yml` regenerates the catalog and deploys the repo to GitHub Pages on
every push to main. The root `index.html` forwards to the builder. On a `*.github.io` origin the
builder auto-detects owner/repo for its remote-include output.
